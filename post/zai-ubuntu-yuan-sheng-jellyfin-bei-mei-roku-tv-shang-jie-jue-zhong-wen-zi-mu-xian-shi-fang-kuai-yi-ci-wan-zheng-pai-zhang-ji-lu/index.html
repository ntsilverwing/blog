在 Ubuntu 原生 Jellyfin + 北美 Roku TV 上解决中文字幕显示方块：一次完整排障记录

2026-06-07

本文一方面给自己做个复盘，避免以后重复踩坑；另一方面也给遇到同类问题的人一个更清晰的排障路径。文中会把“已经实测确认的结论”和“根据官方/社区资料推导出的判断”分开写清楚，尽量避免把经验结论写成绝对事实。

这篇文章记录了一次比较典型、也比较绕的排障过程：Jellyfin Server 跑在 Ubuntu 上，客户端是北美购买的 Roku TV，视频可以正常播放，但一旦加载中文 SRT 字幕，就会显示成方块。最终，这个问题已经基本解决：可以稳定在 Roku TV 上正确显示中文字幕；但现阶段仍不能完全依赖 Roku Jellyfin 客户端直接加载中文 SRT 来显示，更稳定的方案仍然是让服务端参与转码并烧录字幕。

环境与现象

本文对应的环境如下：

  • Jellyfin Server：10.11.10（Ubuntu 原生安装）
  • Jellyfin for Roku：3.20.0
  • 终端设备：北美购买的 Roku TV
  • 字幕类型：外挂中文 SRT

最初的现象非常明确：

  • 视频本身播放正常。
  • 加载中文 SRT 后，字幕显示为方块。
  • Jellyfin Web 端、或服务端参与渲染/烧录时，中文可以正常显示。
  • 只有 Roku Jellyfin 客户端在“自己直接显示中文字幕”这条链路上表现异常。

这意味着问题并不在视频文件损坏，也不太像是 SRT 内容本身有问题，更像是 Roku 端直接渲染中文文本字幕时，拿不到可用的中文字形资源。

先说最终结论

如果只想先拿到可用方案，而不想看排障细节，可以直接看这部分。

已验证可用的方案

这次最终跑通的方案是：

  1. Jellyfin 服务端开启转码相关能力。
  2. 在 Jellyfin 服务端启用备用字体，并把路径指向字体目录，不是单个字体文件。
  3. 使用 .woff 格式的中文字体文件；这次实际跑通使用的是此前在 Windows 环境中验证过的 .woff 字体，并直接复制到 Ubuntu 上继续使用。这个结论是个人实测经验，不是 Jellyfin 官方对所有字体格式的统一结论。
  4. Roku 端在 Settings 中开启全局 Force Transcode；单条视频的 Force Transcode 在这次环境里不稳定，未能可靠生效。
  5. 播放视频后，在当前视频的字幕设置里关闭 Roku 端额外那层字幕显示，只保留服务端已经烧录进视频的中文字幕，避免双层字幕叠加。

按照这套方式，北美 Roku TV 上的中文字幕已经可以正确显示，至少达到了“稳定可看片”的程度。

仍然没解决的部分

虽然最终能正确显示中文，但当前仍然不能完全依赖 Roku Jellyfin 客户端自己去直接显示外挂中文 SRT。 在这次环境下，关闭强制转码、关闭 custom subtitles、只保留单独外挂中文 SRT，依然会出现中文方块，这说明客户端直显中文这条路径并不可靠。

换句话说，现阶段更像是：

  • 服务端烧录中文字幕：可用。
  • Roku Jellyfin 客户端直接显示中文 SRT：不可靠，至少在本文这套环境中不可依赖。

第一个坑：不要把“开启转码”误以为是“已经在转码”

一开始很容易犯的一个错误，是看到日志里出现下面这种信息，就以为当前播放已经走了转码：

EnablePlaybackRemuxing: True
EnableVideoPlaybackTranscoding: True
EnableAudioPlaybackTranscoding: True

但这类日志真正表示的是：当前用户有权限进行 remux / 视频转码 / 音频转码，而不是“这一条播放会话已经发生了转码”。Jellyfin 官方对播放模型的说明很明确：实际是 Direct Play、Direct Stream 还是 Transcode，要由客户端能力、容器、音视频编码和字幕兼容性共同决定。

这一步非常关键，因为如果会话根本没走到服务端字幕渲染，那么你在 Ubuntu 上配置备用字体再正确，也不会对 Roku 直显字幕产生作用。

第二个坑：备用字体路径要指向目录

另一个容易踩的坑，是把 Jellyfin 的“备用字体”直接指向某个具体字体文件，例如：

/opt/jellyfin/roku-fonts/NotoSansCJKsc-Medium.woff

这次排障里，更合理、也最终采用的方式，是把备用字体路径指向字体目录，例如：

/opt/jellyfin/roku-fonts

社区文章和使用经验里，普遍都是按“字体所在目录”来配置，然后由 Jellyfin 在这个目录中查找可用字体。 这一点在 Linux 环境里尤其值得注意，因为很多人直觉上会把路径写成某一个文件，结果导致链路没打通。

第三个坑：这次跑通的是 WOFF，不是“任意字体格式都行”

这里必须把话说严谨。

这次最终跑通的中文显示方案，使用的是此前在 Windows 环境里已经验证过的 .woff 字体文件，复制到 Ubuntu 后继续生效。 但这并不等于已经证明“只有 .woff 能用、.ttf/.otf 一定不行”，因为这次并没有对其它格式做系统化对照测试。

因此，博客里更准确的表述应该是：

  • 本次实测可用的是 .woff 中文字体。
  • 没有进一步验证 .ttf/.otf 在本文环境中是否也能成功。
  • 如果目标是先尽快复现可用方案，优先复用已经验证过的 .woff 是最稳妥的。

第四个坑：Roku 单条视频的 Force Transcode 不一定可靠

Jellyfin 官方在 Roku 3.0.0 的发布说明里提到，可以在单个条目的详情页里开启 Force Transcoding，也可以在用户设置中启用全局强制转码。 从功能设计看，单条视频强制转码本来应该是更方便、更细粒度的开关。

但在这次实测环境里，单条视频的 Force Transcode 并没有稳定生效；只有进入 Settings，把全局 Force Transcode 打开后，中文字幕才真正开始正常显示。 相关 issue 里也有人反馈过类似现象：开启 Force Transcoding 后，客户端行为并不总是完全符合预期。

所以就这次环境来说，更稳妥的做法是：不要依赖单视频开关，直接用全局 Force Transcode。

第五个坑：会出现“双层字幕”

当全局强制转码终于生效后，新的问题又出现了：屏幕上会同时看到两层字幕。

  • 底层是正确显示的中文，说明服务端已经把字幕烧录进视频了。
  • 上层又覆盖了一层方块字幕，说明 Roku 客户端还在把同一条 SRT 当作独立字幕再次渲染。

这一步其实非常有代表性，因为它说明链路已经从“完全没用到服务端字体”变成了“服务端和客户端同时都在处理字幕”。

最终的解决办法并不复杂：

  • 保留全局 Force Transcode；
  • 进入当前视频播放时的字幕设置；
  • 把 Roku 端额外加载的字幕关闭；
  • 只保留服务端烧录进去的那一层字幕。

这样处理后，叠加的异常方块字幕就会消失，只剩下服务端烧录的正常中文字幕。

这次真正定位出的核心原理

排到最后，其实整个问题的关键逻辑已经很清楚了。

1. Roku 端直接显示中文字幕，不依赖 Jellyfin 服务端备用字体

Jellyfin 官方对播放模型的说明是：Direct Play 的目标就是客户端自己处理兼容的音视频和字幕；只有在不兼容时，才由服务端介入进行 Direct Stream 或转码烧录。 这意味着，当 Roku Jellyfin 客户端在本地直接显示外挂 SRT 时，主要依赖的是 Roku 侧这条字幕渲染链，而不是 Ubuntu 服务器上的字体目录。

因此，Jellyfin 服务端备用字体真正起作用的场景，是服务端渲染/烧录字幕，而不是 Roku 本地直接渲染外挂 SRT。

2. custom subtitles 说明 Roku 客户端确实做过 CJK 适配，但这条链路并不稳定

这一点不能简单粗暴地理解成“Roku 完全不支持中文”。

Jellyfin Roku 的历史发布说明里明确提到：

  • 1.6.4 加入了 “Phase 1 CJK subtitle support (External files only)”。
  • 2.0.0 又提到 “Custom subtitle support (required for CJK support) expanded”。

这说明 Jellyfin Roku 客户端确实尝试过针对 CJK 字幕做适配。 也正因为如此，custom subtitles 这个选项并不是摆设，它本来就是为了增强这条字幕显示链。

但问题在于，社区 issue 同样表明，这条链路并不是“开了就一定成功”，而是存在兼容性限制和实现层面的 bug。 在本文这套环境里，单外挂中文 SRT + 关闭转码 + 开/关 custom subtitles 都无法稳定正常显示中文，因此只能得出一个更谨慎但更贴近实测的结论：Roku Jellyfin 客户端当前对中文直显的支持存在限制，至少在北美 Roku TV 上并不可靠。

3. 普通用户没有现实可行的方式去“给 Roku TV 安装中文字体包”

这也是这次排障最后不得不接受的现实。

Roku 开发文档说明，CJK 字体的常见做法是由应用在运行时动态下载并存储到 cachefs/tmp，再由应用自己使用；这属于客户端应用层能力，而不是普通用户能在电视系统设置里完成的事情。 也就是说，普通用户并不能像在 Android TV 或 Linux 上那样，直接往 Roku TV 系统层塞一套中文字体并让官方 Jellyfin App 自动受益。

因此，用户真正能控制的只有两部分：

  • Jellyfin 服务端的字体与转码设置；
  • Roku Jellyfin 客户端已有的几个开关，例如 custom subtitles 与 Force Transcode。

中文元数据 / NFO 中文显示的问题

这里还有一个非常重要、但经常被忽视的问题：不仅字幕，Roku Jellyfin 客户端界面本身的中文也可能显示成方块。

Jellyfin 社区里有过“Chinese library names are displayed as square”的 issue，说明中文库名、中文文本元数据本身就可能出现方块显示的问题。 这类问题本质上和字幕问题很像：都是客户端需要渲染中文字符，但当前显示链拿不到合适的中文字形资源。

因此，如果 NFO、媒体标题、分类名称或其它元数据里包含中文，在 Roku Jellyfin 客户端界面中同样可能无法正常显示中文。 这一点和字幕问题属于同一类限制，只不过一个发生在播放界面，一个发生在应用界面。

需要注意的是，这里不能把结论写得太绝对。更严谨的说法应该是：

  • Roku Jellyfin 客户端存在中文界面文本显示为方块的社区案例。
  • 结合本文的实际测试结果，可以合理推测：如果 NFO/元数据大量使用中文，Roku 端界面同样可能出现方块。
  • 这说明问题不仅是“字幕显示不全”，而是当前客户端整体的 CJK 文本渲染能力有限。

一套更实用的最终建议

如果目标不是继续研究原理，而是以后稳定看片，那么更实用的建议是这样的：

场景 建议做法 原因
使用 Roku TV 看中文内容 开全局 Force Transcode，服务端烧录中文字幕 已被本文环境实测验证可用。
播放后出现两层字幕 在当前视频里关闭 Roku 端额外字幕显示 避免客户端再叠加一层方块字幕。
Jellyfin 服务端字体配置 备用字体指向字体目录，优先复用已验证可用的 .woff 中文字体 这是本文最终跑通的配置方式。
想让 Roku 直接显示外挂中文 SRT 可以尝试，但不要把它当成可靠方案 本文环境下这条路径不稳定。
NFO / 界面元数据大量使用中文 预期 Roku 客户端可能仍显示方块 已有中文文本显示为方块的社区案例。

这次排障最大的经验

回头看，这次排障最浪费时间的地方，并不是“没找到开关”，而是一开始没有把“客户端渲染”和“服务端渲染”这两条链路分开理解。

一旦把这件事想清楚，很多现象都会变得非常合理：

  • 服务端备用字体为什么在某些情况下完全没效果，因为那时字幕根本没有经过服务端渲染。
  • 为什么强制转码后中文突然正常，因为这时真正走到了服务端烧录字幕。
  • 为什么又会冒出第二层方块字幕，因为客户端还在重复渲染同一条字幕。
  • 为什么不仅字幕，中文元数据也可能出问题，因为 Roku 客户端界面本身对中文文本渲染也有限制。

结语

到目前为止，这个问题可以说是“工程上已经解决”，但“产品体验上仍不完美”。北美 Roku TV 上的 Jellyfin 已经可以通过服务端烧录的方式正确显示中文字幕，但还不能把“客户端自己直接显示中文 SRT”当成稳定方案来依赖。

如果后续 Jellyfin Roku 客户端继续完善 CJK 字幕链路，或者对中文界面文本支持更完整，那么这套方案未来还有继续简化的可能。 但至少在 Jellyfin Server 10.11.10 + Roku Client 3.20.0 + 北美 Roku TV 这套环境里，当前最稳的现实方案仍然是：服务端准备好字体，Roku 端开启全局强制转码，并关闭重复叠加的客户端字幕。
