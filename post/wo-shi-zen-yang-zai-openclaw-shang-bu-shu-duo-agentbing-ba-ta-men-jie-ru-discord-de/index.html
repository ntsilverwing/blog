我是怎样在 OpenClaw 上部署多 Agent，并把它们接入 Discord 的

2026-04-18 openclaw ， AI

这不是一篇"照着官方文档抄配置"的教程，而是一篇完整的实践复盘。
我从单 Agent 出发，最终在 Ubuntu 24 VPS 上搭出了一套可运行的 OpenClaw 多 Agent系统，并完成了 Discord 集成。过程中最有价值的部分，不是最后那份配置文件，而是中间几次关键认知修正。

一、我为什么要折腾多 Agent

一开始，我的目标很明确：我不想让一个大而全的 Agent 既负责聊天、又负责搜索、还负责写代码和运维。这样虽然"能用"，但任务一复杂，角色就会混乱，模型成本和上下文负担也会一起上升。

所以我决定把系统拆成三个角色：

  • 主脑 mainbrain：负责和我对话、理解任务、拆解任务、调用其他 Agent、汇总最终结果
  • 外脑 MAGI：负责搜索、调研、长文档分析、补充思路和提供外部参考
  • 码农 codefarmer：负责工具调用、代码编写、自主测试、排错和运维执行

从结果上看，这个方向是对的。真正难的不是"定义三个名字"，而是让这三个 Agent 在 OpenClaw 的实际机制里稳定协作起来。

二、最初的错误直觉：让 Discord 里的 Bot 互相 @

我最早的设想其实很自然：既然我已经在 Discord 里配了多个 bot，那是不是可以让主脑 bot 在频道里直接 @MAGI 或 @codefarmer，把子任务派出去，等它们做完再 @ 主脑回报结果？

但这个思路后来被我彻底放弃了。原因不是"不优雅"，而是不稳定：在 OpenClaw 的多 Bot 场景里，bot 消息过滤、allowBots/respondToBots 行为的覆盖冲突、共享频道造成的上下文污染，以及最危险的 Ping-Pong 死循环，都会让"Bot 互相 @"变成一个很脆弱的方案。

我后来确认下来的结论是：

  • Discord 更适合做人类可见的 UI 层，不适合作为 Agent 之间的真实通信总线
  • Agent 之间的协作，应该走 OpenClaw 内建的 sessions_send / sessions_spawn
  • 如果一定要在 Discord 里做可视化过程展示，也应该让它只承担"展示"和"旁路通知"的职责，而不是让 Bot 彼此靠频道消息驱动

这一步认知变化，基本决定了后面整套架构会不会稳。

三、真正可行的通信方式：Discord 做 UI，Session 工具做调度

搞清楚这个问题之后，整个系统的理解就清楚了：用户在 Discord 里跟主脑对话，主脑在底层通过 OpenClaw 的 session 机制调度其他 Agent，子 Agent 的结果再回到主脑，由主脑统一对外输出。

这里有两个关键工具：

sessions_spawn：适合一次性子任务

如果主脑需要临时派发一个独立任务，比如"去调研某个方案""分析这份文档""生成一份报告"，那更适合用 sessions_spawn。它会创建一个新的隔离 session，子任务在后台跑，完成后再回传结果。

这类任务很像"开一个子进程"。配合 thread: true 参数使用，复杂任务的过程会折叠进 Discord Thread，这样既能监工，又不会把主频道刷爆。

// 主脑调用示例
sessions_spawn({
  task: "请调研最新的前端框架并总结要点",
  label: "research-task-01",
  agentId: "magi",
  runTimeoutSeconds: 120,
  thread: true   // 复杂任务推荐加这个，过程折叠进 Thread
})

执行后立即返回 { status: "accepted", runId, childSessionKey }，子任务在后台跑，完成后结果自动回传主脑。

sessions_send：适合和常驻 Agent 持续协作

如果目标 Agent 本来就是常驻的，比如 codefarmer 长期负责同一个代码库的调试，那更适合用 sessions_send。它不是新建一次性子任务，而是给已有 session 发消息，可以进行有限轮次的来回交流。

sessions_send({
  sessionKey: "agent:codefarmer:main",
  message: "帮我 review 这段代码并给出修改建议",
  timeoutSeconds: 60   // 等待 60 秒回复
})

一次性任务和持续协作在 OpenClaw 里是两套不同机制，理解这个区别非常重要。

四、从"能跑"到"正确"：单 Gateway 多 Agent 的几个硬坑

坑一：一个 Agent 必须对应一个独立 Bot Token / accountId

单 Gateway 多 Agent 并不是"一个 Telegram Bot / 一个 Discord Bot 然后路由给多个 Agent"，而是每个 Agent 都应该有独立的 bot 账号和独立 accountId，然后通过 bindings 绑定到对应 agent。

channels: {
  telegram: {
    accounts: {
      default:    { botToken: "TOKEN_FOR_MAINBRAIN" },
      magi:       { botToken: "TOKEN_FOR_MAGI" },
      codefarmer: { botToken: "TOKEN_FOR_CODEFARMER" },
    },
  },
  discord: {
    accounts: {
      default:    { token: "DISCORD_TOKEN_MAINBRAIN" },
      magi:       { token: "DISCORD_TOKEN_MAGI" },
      codefarmer: { token: "DISCORD_TOKEN_CODEFARMER" },
    },
  },
},

这点尤其影响 Discord。因为在同一个 guild 里，如果只是写多条 binding 而不区分 accountId，最后往往只有最后一条生效，表面上像"配置写了"，实际上路由是错的。

坑二：agentDir 和 workspace 不是一回事

OpenClaw 里每个 Agent 还有独立的 agentDir（存放 auth profiles、model registry 等内容），绝对不能多个 Agent 共用，否则 session 和认证很容易冲突。

真正的多 Agent 隔离，不只是"每人一个工作目录"，而是"每人一套 agentDir + workspace + accountId"三者全部独立。

# agentDir 由 openclaw agents add 自动创建，无需手动创建
# 但如果多个 Agent 需要共享同一个 API 凭证，需手动复制：
cp ~/.openclaw/agents/mainbrain/agent/auth-profiles.json \
   ~/.openclaw/agents/magi/agent/auth-profiles.json

坑三：从单 Agent 迁移不能硬改，要平滑过渡

我的环境不是从零开始的，而是已经跑通了 Ubuntu 24 VPS + CLI 安装 + Node.js v22+ + TG 和 Discord 的单机器人版本。

所以更稳的迁移路径是：

# 1. 先备份旧配置
cp ~/.openclaw/openclaw.json ~/.openclaw/openclaw.json.bak

# 2. 把原有 workspace 改名为 workspace-mainbrain
mv ~/.openclaw/workspace ~/.openclaw/workspace-mainbrain

# 3. 在配置中显式声明现有 bot 为 main agent 的默认账号

# 4. 先确认原有功能不中断，再逐步添加新 agent 和 bindings
openclaw gateway restart

这个迁移视角，比"新手从零搭建教程"更接近真实生产环境。

坑四：Ubuntu VPS 上一定要上 systemd

只靠 openclaw gateway start 远远不够。SSH 断开、进程异常退出、服务器重启，都会导致服务中断。对外看，补一个 systemd service 文件只是"小操作"，但它是"玩具系统"和"长期运行系统"的分水岭：

# /etc/systemd/system/openclaw.service
[Unit]
Description=OpenClaw Gateway
After=network.target

[Service]
Type=simple
User=youruser
WorkingDirectory=/home/youruser
ExecStart=/usr/local/bin/openclaw gateway
Restart=on-failure
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target

sudo systemctl daemon-reload
sudo systemctl enable --now openclaw
sudo systemctl status openclaw
sudo journalctl -u openclaw -f

五、模型分工：不是"哪个强用哪个"，而是按角色分配

当架构稳定之后，开始认真处理另一个问题：三个 Agent 应该分别使用什么模型。

这次我结合了模型能力、上下文窗口大小和 OpenClaw 的实际配置格式，整理出一套更像"工程分工"而不是"排行榜崇拜"的分配思路。

主脑：优先规划与调度能力

主脑不是拿来刷 benchmark 的，它最重要的是任务分解、长链路规划、输出结构化结论。核心思路是：

  • 主力模型承担全局规划（高输出 token 限制是优势）
  • 超长上下文模型作为长会话接替（超过上下文阈值时接棒）
  • 轻量模型负责高频短路由，减少主力 token 压力
// 主脑模型配置示意（OpenClaw agents.list 格式）
{
  id: "mainbrain",
  model: {
    primary: "z-ai/glm5",
    fallbacks: [
      "qwen/qwen3.5-397b-a17b",          // 超长会话接替：1M 上下文
      "openrouter/z-ai/glm-4.5-air:free", // 轻量降级路由
      "openrouter/openai/gpt-oss-120b:free",
    ],
  },
  thinkingDefault: "low",
  timeoutSeconds: 900,
}

外脑：搜索执行和内容消化要分开看

外脑不是"会搜就行"。真正的外脑要分成两层：前一层负责搜索执行和资料召回，后一层负责把大量材料吃进去做归纳整理。 有的模型更适合出去找资料，有的模型更适合消化超长上下文。把这两件事分开，外脑才真像"智库"而不是"高级搜索框"。

{
  id: "magi",
  model: {
    primary: "moonshotai/kimi-k2.5",      // 262K，并行搜索执行
    fallbacks: [
      "minimaxai/minimax-m2.5",            // 1M 上下文，大批量内容消化
      "meituan-longcat/LongCat-Flash-Thinking-2601", // 深度推理验证
      "meituan-longcat/LongCat-Flash-Lite", // 快速召回预筛选
    ],
  },
  thinkingDefault: "low",
  timeoutSeconds: 600,
}

码农：执行效率比"全能"更重要

码农的职责很明确：能调工具、能写代码、能看报错、能连续修，比"会讲道理"更重要。thinkingDefault 设为 off，优先保证执行速度；timeoutSeconds 设为 1800，因为运维任务天然耗时更长。

{
  id: "codefarmer",
  model: {
    primary: "openrouter/stepfun/step-3.5-flash:free", // SWE-bench 74.4%，免费
    fallbacks: [
      "minimaxai/minimax-m2.5",
      "meituan-longcat/LongCat-Flash-Chat",
      "openrouter/qwen/qwen3-coder:free",  // 仓库级代码理解，256K
    ],
  },
  thinkingDefault: "off",
  timeoutSeconds: 1800,
}

六、Prompt 文件结构：规则为什么要拆开

工作区文件怎么组织，看起来是"文案整理"，实际上是架构问题。

我中间有一段专门讨论：主脑的 AGENTS.md 里，到底要不要把"我是 Commander，我有哪几个下属，怎么协作"全都写进去。

最后接受的做法是拆成三层：

文件 职责
AGENTS.md 角色入口、会话启动规则、"先读哪些文件"的引导
ROLE-COLLAB-RULES.md 团队分工和协作流程，统一收口
TOOLS.md 工具使用边界和注意事项，包含跨 Agent 调用规范

这样做的好处很实际：避免同一规则在多个文件里重复（浪费 token）；以后改团队结构时只改一处；把"角色认知"和"工具避坑"拆开后，Agent 的行为更稳。

主脑 AGENTS.md 示例（精简入口版）：

# AGENTS.md - Your Workspace

This is your central command center.

## Every Session
1. Read `SOUL.md` — this is who you are.
2. Read `USER.md` — this is who you're serving.
3. Read `memory/YYYY-MM-DD.md` (today + yesterday) for recent context.
4. Read `MEMORY.md` for long-term curated memories.

## Swarm Orchestration
You are the **Commander**. You do not work alone.
Before executing complex code or deep research, you MUST read
`ROLE-COLLAB-RULES.md` to understand your team (@magi, @codefarmer)
and how to delegate tasks to them.

七、Discord 频道结构：最终定下来的布局

在 Discord 集成上，最终不再追求"Bot 之间在公开频道里演戏"，而是让频道职责更清晰：

频道 谁在里面 用途
#主脑控制台 我 + mainbrain 主入口，下发任务，看最终汇总结果
#研究专区 我 + MAGI 直接向 MAGI 提问，或查看研究过程
#代码车间 我 + codefarmer 直接派代码任务，查看报错和修复过程
#系统通知 我 + 三个 Bot（只写不读） 全局进度看板，Bot 主动推送关键状态

关键权限原则：#系统通知 频道所有 Bot 关闭 View Channel History，只保留 Send Messages，防止 Bot 读取彼此的状态消息然后互相触发。

一次完整任务的工作流：

你（在 #主脑控制台）
↓ "帮我调研最新前端框架，写一个接入 Demo"

mainbrain
↓ [底层 sessions_spawn, thread:true] → #主脑控制台 自动开 Thread
↓ 系统通知: "🧠 mainbrain: 任务拆解中..." → #系统通知

MAGI（在 Thread 里直播搜索过程）
↓ 系统通知: "🔬 MAGI: 开始调研..." → #系统通知
↓ 系统通知: "🔬 MAGI: ✅ 调研完成" → #系统通知
↓ [结果自动回传 mainbrain]

mainbrain
↓ [底层 sessions_send → codefarmer]

codefarmer（同样在 Thread 里输出代码过程）
↓ 系统通知: "💻 codefarmer: ✅ 完成，改了4个文件" → #系统通知
↓ [结果回传 mainbrain]

mainbrain
↓ 系统通知: "🧠 mainbrain: 汇总完成" → #系统通知
↓ 在 #主脑控制台 输出最终汇总报告

你（在 #主脑控制台 看到干净的最终结果，需要"监工"时点开 Thread）

八、部署收益的实际数据

经过这一轮部署，实际记录了两个典型案例的运行数据：

指标 数值 说明
平均响应时间 2–5 秒 取决于任务复杂度
Token 消耗优化 40–60% 通过按需加载和角色分工
系统可用性 99.5%+ 通过 fallback 机制和 systemd 守护

案例一：多 Agent 协作研究 OpenClaw Agent 人格 MD 按需加载优化方案
总耗时约 8 小时，产出 7 份设计文档，Token 消耗比单 Agent 方式下降约 40–60%。

案例二：HDD 坏道分析
mainbrain 接收需求、指派给 codefarmer，codefarmer 读取检测文件、分析坏道分布、生成分区建议，mainbrain 审核并纠正逻辑错误，全流程耗时约 5 分钟。

九、我最终理解的 OpenClaw 多 Agent

走完这一轮之后，对 OpenClaw 多 Agent 的理解发生了很大变化。它不是"给三个 Bot 配三个模型"这么简单，而是一个同时包含路由、隔离、协作协议、工作区设计、可视化交互的系统工程。

如果只看最终结果，这套系统可以概括成一句话：

一个主脑负责对外，一个外脑负责信息，一个码农负责执行；
Discord 是面板，Session 是总线，Workspace 文件是治理层。

十、给后来者的几条经验

如果你也准备自己搭一套 OpenClaw 多 Agent，这几条最值得先知道：

  • 不要先从"Bot 怎么互相 @"开始想——先把 Agent 间通信和 Discord UI 分层
  • 不要偷懒复用 bot / accountId / agentDir——多 Agent 的隔离是基础，不是优化项
  • 不要只关注模型强弱——先明确每个 Agent 的职责，再去选模型
  • 不要把所有规则都塞进 AGENTS.md——角色、协作、工具边界最好拆分维护
  • 不要忽略长期运行问题——Ubuntu VPS 上的 systemd、备份、迁移策略都应该提前考虑

结语

这次部署最大的收获，不是"我把 OpenClaw 跑起来了"，而是终于把"多 Agent 系统到底该怎么落地"这件事想明白了。

从最开始想让 Bot 在 Discord 里彼此 @，到后来改用 sessions_spawn / sessions_send；从单一机器人，到每个 Agent 独立 bot、独立账号、独立目录；从 prompt 堆砌，到分层治理文件；从泛泛模型列表，到角色化模型分工——这整个过程本身，才是这篇文章真正值得分享的内容。
